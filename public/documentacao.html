🔐 Autenticação (DASH_TOKEN)

Se DASH_TOKEN estiver definido no .env, os endpoints /events, /send, /messages, /chats exigem token:

  • Por cabeçalho: Authorization: Bearer <DASH_TOKEN>
  • Ou por query string em SSE: /events?token=<DASH_TOKEN>

Se DASH_TOKEN não estiver definido, a API permanece aberta (somente para desenvolvimento).

📡 SSE — GET /events

Stream de eventos via Server-Sent Events. É por aqui que chegam: status, qr (como DataURL) e message (payloads de mensagens). 

// Exemplo (browser):
const token = 'SEU_DASH_TOKEN_OPCIONAL';
const es = new EventSource(token ? `/events?token=${encodeURIComponent(token)}` : '/events');

es.addEventListener('status', (e) => {
  const data = JSON.parse(e.data);
  // stages: connected | loading | authenticated | ready | disconnected | warn | error
  console.log('status', data);
});

es.addEventListener('qr', (e) => {
  const data = JSON.parse(e.data);
  // data.qrDataURL (img base64) ou { qr: 'texto' } como fallback
  console.log('qr', !!data.qrDataURL);
});

es.addEventListener('message', (e) => {
  const msg = JSON.parse(e.data);
  // ver esquema em "Esquema de mensagem" abaixo
  console.log('message', msg);
});
EventoDescriçãoExemplo
status Fases do cliente WhatsApp (loading, authenticated, ready…), warnings e erros. {"stage":"ready"}
qr QR Code para parear o WhatsApp. Preferir qrDataURL (imagem base64). {"qrDataURL":"data:image/png;base64,..."}
message Mensagens enriquecidas (texto, imagem, sticker, áudio, vídeo, docs…). Ver “Esquema de mensagem”.
ping Heartbeat automático para manter a conexão aberta. timestamp numérico

🖼️ Serviço de mídia

Ao receber mídia, o backend salva no disco em MEDIA_DIR e expõe:

  • GET /media/<id>.<ext> — estático (com Content-Type correto)
  • GET /media/by-id/:id — fallback quando você só conhece o id
<!-- Exibição no front (ordem recomendada): -->
<img src="<dataUrlPreview> || <mediaUrl> || /media/by-id/<id>" />

<audio controls src="<mediaUrl> || /media/by-id/<id>"></audio>

<video controls src="<mediaUrl> || /media/by-id/<id>"></video>
Obs.: stickers chegam como imagem (ex.: image/webp), e PTT como áudio (audio/ogg).

✉️ Envio — POST /send

Envia uma mensagem de texto para um chat ou contato.

CampoTipoObrigatórioDescrição
tostringsimJID do chat. Ex.: 5561XXXXXXXX@c.us (contato) ou XXXXXXXX@g.us (grupo)
bodystringsimTexto da mensagem
 
# Com token (se configurado)
curl -X POST http://localhost:10000/send \
  -H 'Authorization: Bearer SEU_TOKEN' \
  -H 'Content-Type: application/json' \
  -d '{"to":"5561999999999@c.us","body":"Olá!"}'
// fetch no browser
await fetch('/send', {
  method: 'POST',
  headers: {'Content-Type':'application/json','Authorization':'Bearer SEU_TOKEN'},
  body: JSON.stringify({ to: '5561999999999@c.us', body: 'Olá!' })
});
Dica: no front já existe “bolha otimista” para evitar duplicação quando o message_create chegar via SSE.

📜 Histórico — GET /messages

Lista mensagens persistidas (MongoDB). Somente mensagens “normais” (broadcast/status são ignoradas na ingestão).

QueryTipoDefaultDescrição
limitint50 (máx 500)Quantidade de documentos
chatIdstring—Filtra por chatId
days / sinceDaysint—Janela deslizante de N dias (filtro de leitura; não apaga)
 
curl -H 'Authorization: Bearer SEU_TOKEN' \
"http://localhost:10000/messages?limit=100&days=7"

🗂️ Chats — GET /chats

Lista agregada de conversas (última mensagem por chat).

✅ Healthcheck — GET /healthz

Retorna ok quando o servidor HTTP está respondendo.

  "direction": "in|out",                // baseado em fromMe
  "chatId": "XXXXXXXXXX@c.us|@g.us",

  "to": "jid de destino",

  "ts": 1729960000000,                  // timestamp em milissegundos

  "mimetype": "image/webp|audio/ogg|...",
  "filename": "nome original (quando houver)",
  "size": 12345,                        // bytes (se baixou)

Observações: status@broadcast é ignorado na ingestão. Stickers chegam como imagem (geralmente image/webp). PTT vem como áudio (audio/ogg).

🚀 Guia de integração rápida

  1. Abra um canal SSE em /events (com token, se houver) e:
    • Renderize qr.qrDataURL em um <img> para parear.
    • Espere status.stage === "ready".
  2. Monte sua UI ouvindo o evento message. Para cada mensagem:
    • Se type começa com imagem/vídeo/áudio, use mediaUrl (ou /media/by-id/:id).
    • Para sticker, trate como imagem.
    • Para PTT/áudio, use <audio controls src="...">.
  3. Para enviar texto, use POST /send com {to, body}.
  4. Para popular histórico/contatos:
    • GET /messages?limit=50&days=7
    • GET /chats?days=30
// Exemplo mínimo (browser)
const token = 'SEU_DASH_TOKEN';
const es = new EventSource(`/events?token=${encodeURIComponent(token)}`);
es.addEventListener('qr', e => {
  const d = JSON.parse(e.data);
  document.querySelector('#qr').src = d.qrDataURL;
});
es.addEventListener('message', e => {
  const m = JSON.parse(e.data);
  // renderize conforme m.type (chat/image/sticker/audio/video/document)
});

// Envio:
await fetch('/send', {
  method:'POST',
  headers: {'Content-Type':'application/json','Authorization':'Bearer '+token},
  body: JSON.stringify({ to:'5561999999999@c.us', body:'Olá!' })
});
Dica: já há deduplicação de mensagens no front usando “bolhas otimistas” e conciliação quando o message_create chega.

🧭 Notas importantes

  • Mensagens de status@broadcast são ignoradas na ingestão.
  • Se MSG_TTL_DAYS > 0, a coleção cria TTL por createdAt (limpeza automática no Mongo).
  • Com REDIS_URL, a persistência é feita em lotes a cada FLUSH_SECONDS.
  • Ao rodar no Render, use PUPPETEER_EXECUTABLE_PATH=/usr/bin/chromium e PUPPETEER_SKIP_DOWNLOAD=true (já contemplado no Dockerfile).
